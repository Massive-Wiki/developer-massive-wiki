<h1>Tech Note - Preferring a Different Git Forge</h1> <p><em>last edit: 2022-12-06, Pete</em></p> <h2>Overview</h2> <p>Up until now (~2022-12), Massive Wiki community projects that use a <span class="incipient-wikilink">Git forge</span>, either for code, or for syncing and versioning wiki files, have typically used <span class="incipient-wikilink">GitHub</span> as their centralizing forge, for the sake of convenience and audience reach.</p> <p>However, a key principle of Massive Wiki, and wiki in general, is enablement and encouragement of <span class="incipient-wikilink">open collaboration</span>. While GitHub was a key catalyzer of the open source movement, the software underlying the platform has never been open source. Also, Microsoft's purchase and control of GitHub since 2018 is problematic for the open source ecosystem.</p> <p>Therefore, the Massive Wiki community wishes to reduce its GitHub footprint, and increase use of another Git forge. This page will serve to illuminate some of the choices and challenges in deciding which Git forge (or forges) to migrate to.</p> <h2>Capabilities and Limitations of Forges</h2> <p>An initial observation: because Git is already a standardized protocol for moving files in a repo from one computer to another, migrating between forges is easy and, in fact, is similar to normal use of Git.</p> <p>However, there <em>are</em> differences between forges in the collaboration and operations (e.g., deployment) processes they enable.</p> <p>For instance, GitHub has easy and rich features that allow discussion and workflow around Git forks, branches, and commits. For the deployment of static web sites (such as those built by <span class="incipient-wikilink">Massive Wiki Builder</span>), <span class="incipient-wikilink">Netlify</span> and <span class="incipient-wikilink">Vercel</span> can automate deployment for repos hosted by GitHub, GitLab, and BitBucket.</p> <p>In moving to another forge, the Massive Wiki community will need to adapt its workflows to the capabilities and limitations of the new forge.</p> <h2>SaaS vs. Self-Hosted</h2> <p>A key consideration in evaluating a new forge is whether the forge will be a cloud SaaS platform operated by a third party, or if it will be self-hosted by Massive Wiki community members or partners.</p> <h3>SaaS</h3> <p>Cloud SaaS platforms include GitHub, GitLab, BitBucket, SourceForge, SourceHut, and Codeberg.</p> <p>Our short list for open collaboration includes GitLab, SourceHut, and Codeberg.</p> <h3>Self-Hosted</h3> <p>Software for self-hosted Git forges include <a href="https://about.gitlab.com/install/">GitLab</a>, <a href="https://gitea.io/en-us/">Gitea</a>, <a href="https://forgejo.org/">Forgejo</a>, <a href="https://gogs.io/">Gogs</a>, <a href="https://www.gerritcodereview.com/">Gerrit</a>, <a href="https://man.sr.ht/installation.md">SourceHut</a>, and <a href="https://gitolite.com/">Gitolite</a>.</p> <p>GitLab, SourceHut, and Gerrit require significant infrastructure setup.</p> <p>Gitolite is great for a personal Git forge, but is too lightweight for a collaborative forge.</p> <p><a href="https://blog.gitea.io/2016/12/welcome-to-gitea/">Gogs was forked in 2016</a> to create Gitea. Gitea is likely a better choice.</p> <p>The <a href="https://blog.gitea.io/2022/10/open-source-sustainment-and-the-future-of-gitea/">Gitea maintainers announced a commercial company, Gitea Ltd</a> in 2022-10, which now owns the Gitea domains and trademarks and stewards the open source project.</p> <p>This <a href="https://news.ycombinator.com/item?id=33749757">spooked some of the Gitea community</a>, particular Codeberg, which <a href="https://forgejo.org/faq/">forked Gitea to create Forgejo</a> in 2022-11. As of this writing, it's not clear whether Gitea or Forgejo would be a better choice for a self-hosted forge.</p> <h2>Collective Sense Commons</h2> <p>Peter Kaminski, maintainer of Collective Sense Commons (CSC), has expressed interest in having CSC host a Gitea (or now, perhaps Forgejo) instance. CSC and Massive Wiki are partner communities.</p> <h2>Current Thinking by Pete</h2> <h3>SourceHut</h3> <p>I have a ton of respect for Drew DeVault, SourceHut's creator and maintainer, and I've been a paid personal SourceHut supporter at the "Amateur Hackers" level since 2018-11.</p> <p>Drew is a staunch open source advocate. He runs SourceHut in an opinionated and focused manner, and I appreciate the strength of his leadership and his dedication to transparency and community service. However, since Massive Wiki repos are more often content than code, I am concerned that SourceHut is not an appropriate venue as a general purpose forge for Massive Wiki.</p> <h3>CSC Gitea or Forgejo</h3> <p>I would love to have CSC host a Git forge for Massive Wiki and other communities.</p> <p>CSC currently does not have the resources or bandwidth it would require, though.</p> <h3>Codeberg, and the Need for CI / Build</h3> <p>Therefore, I think Codeberg may be the best forge to focus on for the Massive Wiki community at this time.</p> <p>We currently rely heavily on automated deployment from GitHub to Netlify for the Massive Wiki community, and it enables non-technical users of Massive Wiki to publish to the web, without any manual intervention (after starting with the Massive Wiki Starter repo template).</p> <p>Netlify has API access, and I've used it to publish static sites. It appears, though, that Netlify won't build a site that's uploaded via API.</p> <p>A service could be built that takes a webhook notification from Codeberg, builds the Massive Wiki static files, then uploads them to Netlify. This could be Massive Wiki-specific (it only builds Massive Wikis), or more general, by using a CI tool like <a href="https://woodpecker-ci.org/">Woodpecker CI</a>.</p> <p>Another thing to check: whether Vercel would do a build and deploy via API.</p> <h2>Continuing Evolution</h2> <p>Preferring a particular forge for now -- or even preferring a forge, rather than something that works peer-to-peer -- is not an end state, but another step in an evolution.</p> <p>The community should continue to evaluate its Git coordination strategy, and because it's Git, it should never be too hard to make another step towards a preferred solution.</p>
