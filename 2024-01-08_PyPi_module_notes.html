<h1>2024-01-08 PyPI module notes</h1> <p>(note by Bill)</p> <ul> <li> <p>reviewed several PyPI module build tools and settled on two prospects:</p> <ol> <li><code>flit</code>: simple to put into practice; minimal metadata required</li> <li><code>hatch</code>: also simple; metadata seems more complicated; also supports more options</li> </ol> </li> <li> <p>(also experimented with using <code>build</code> and <code>twine</code> modules: very simple; does not support dependencies so that is not suitable for Massive Wiki publishing.)</p> </li> <li> <p>important to create and use <code>test.pypi.org</code> to work out the use details; requires creating a login and an API key</p> </li> <li> <p>reading the <code>flit</code> documentation, care must be taken to insure that testing uses the <code>test.pypi.org</code> system; seems easy to end up using the default, which is the main PyPI server, and that needs to be avoided.</p> <ul> <li>see how to upload to <code>test.pypi.org</code> here: <a href="https://flit.pypa.io/en/stable/upload.html#using-pypirc">Using .pypirc, Controlling package uploads</a></li> </ul> </li> <li> <p>questions to answer:</p> <ul> <li>best to just have an empty <code>__init.py__</code> file?</li> <li>where is a good place to define the <code>version</code> descriptor? (reason for this question: some docs show it in the <code>pyproject.toml</code> file which seems like a terrible place -- once all that metadata is set up just leave that file alone; some docs show it imported into the <code>__init.py__</code> file, but how is that used?; <code>flit</code> wants the version descriptor at the top of the source file -- OK, that seems maintainable )</li> </ul> </li> </ul> <hr /> <h2>Resources</h2> <ul> <li> <p>Flit: <a href="https://flit.pypa.io/en/stable/index.html">Flit 3.9.0 — Flit 3.9.0 documentation (pypa.io)</a>)</p> </li> <li> <p>Hatch: <a href="https://hatch.pypa.io/1.9/">About - Hatch (pypa.io)</a></p> </li> <li> <p>Twine: <a href="https://pypi.org/project/twine/">twine · PyPI</a></p> </li> <li> <p>Build: <a href="https://pypi.org/project/build/">build · PyPI</a></p> </li> <li> <p>An example of a good documentation practice to emulate: <a href="https://towardsdatascience.com/the-complete-guide-for-creating-a-good-pypi-package-acb5420a03f8">The Most Complete Guide for Creating a Good PyPI Package | by Elise Landman | Towards Data Science</a></p> <ul> <li>what I like about this description is the inclusion of the barebones project directory tree and the warm tone and style of the writing.</li> </ul> </li> </ul>
