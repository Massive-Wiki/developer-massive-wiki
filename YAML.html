<h2>Description</h2> <p>"YAML is a human friendly data serialization standard for all programming languages." - <a href="https://yaml.org/">yaml.org</a></p> <h2>Guides</h2> <p>The Camel <a href="https://camel.readthedocs.io/en/latest/yamlref.html">Brief YAML reference </a></p> <h2>Specification</h2> <p><a href="https://yaml.org/spec/1.2/spec.html">YAML Ain’t Markup Language (YAML™) Version 1.2</a></p> <h2>Notes, Tips</h2> <h3>Regarding Markdown "Front Matter"</h3> <p>"<code>---</code>" separates sections ("documents") of YAML. Using "<code>---</code> (YAML lines) <code>---</code>" for Markdown frontmatter is sort of a hack; "<code>---</code>" in Markdown will mean a horizontal line, and seeing "<code>---</code> (YAML lines)" is YAML-friendly. By definition, "<code>---</code>" will never appear in a YAML "document" (since it delimits the start of the next document), so it's safe to use the second "<code>---</code>" to delimit the end of first YAML document -- er, the Markdown front matter.</p> <p>(In YAML, though, the real end of a document is "<code>...</code>". So if there's no "<code>...</code>" at the end of the front matter, a naive YAML parser reading a Markdown file with front matter would expect the Markdown to be a second YAML document, so don't feed your Markdown file with front matter to a naive YAML parser.)</p> <p>(Ironically <em>(in the Morrisette sense)</em>, YAML has its own "front matter"; in a YAML file, before the first "<code>---</code>", there can be optional "directives". Markdown-with-front-matter doesn't typically have YAML directives, though.)</p>
